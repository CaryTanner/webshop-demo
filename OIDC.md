# OIDC Authentication Design

This document describes a high-level design for implementing authentication using **OpenID Connect (OIDC)** in a web application with a frontend, backend services, and an external identity provider.

The solution uses the **Authorization Code Flow with PKCE** together with a **Backend-for-Frontend (BFF)** pattern to provide strong security while keeping the frontend implementation simple.

---

## 1. How the Frontend Authenticates via OIDC

When the user clicks **Login**, the frontend redirects the browser to the login page of the OpenID Provider (OP).

The application first retrieves configuration from the OIDC discovery endpoint:

This endpoint provides important information such as:

- authorization endpoint
- token endpoint
- JWKS endpoint (used for token verification)

The authorization request includes parameters such as:

- `response_type=code`
- `client_id`
- `redirect_uri`
- `state` (protects against CSRF attacks)
- `nonce` (prevents ID token replay)
- `code_challenge` (used for PKCE)

The `code_challenge` is derived from a random **code verifier** generated by the client.

After the user authenticates (for example via password, SSO, MFA, or social login), the OP redirects the user back to the application with an **authorization code**.

The backend then exchanges this authorization code at the token endpoint together with the **code verifier** to obtain tokens.

The OP typically returns:

- **ID Token** – proves that the user authenticated successfully and contains identity claims about the user. It is intended for the client application.
- **Access Token** – used by backend services to authorize requests to protected APIs.
- **Refresh Token** – used to obtain new access tokens when the current one expires.

---

## 2. How Tokens Are Handled

In this design, tokens are managed by a **Backend-for-Frontend (BFF)** service rather than the browser.

The BFF exchanges the authorization code for tokens and stores them in a **server-side session store** (for example Redis or DynamoDB). The browser receives a **secure HttpOnly session cookie** that identifies the user session.

This approach avoids exposing tokens to the browser, reducing risks such as token theft through XSS.

When the browser makes requests to the application, it sends the session cookie to the BFF. The BFF retrieves the corresponding tokens from the session store and uses the **access token** when calling backend services.

If the access token expires, the BFF can use the **refresh token** to obtain a new access token without requiring the user to log in again.

---

## 3. How the Backend Verifies Tokens

Backend services verify incoming access tokens before processing requests.

This typically involves:

1. Verifying the **JWT signature** using the public keys published by the OpenID Provider's **JWKS endpoint**
2. Validating standard claims such as:
   - `iss` (issuer)
   - `aud` (audience)
   - `exp` (expiration)
3. Optionally validating authorization-related claims such as scopes or roles.

Public keys from the JWKS endpoint are typically **cached and periodically refreshed** to avoid unnecessary network calls and improve performance.

This allows each microservice to independently determine whether a request is authorized.

## 5. Security and Design Considerations

Several security considerations influence this design:

### PKCE

PKCE protects the authorization code exchange and prevents authorization code interception attacks.

### Token Storage

Using a BFF prevents tokens from being stored in the browser (for example in localStorage), which reduces the risk of token theft through XSS.

### Session Cookies

Cookies should be configured with:

- `HttpOnly`
- `Secure`
- `SameSite`

to minimize exposure to browser-based attacks.

### Microservice Authorization

Each backend service verifies access tokens independently, allowing services to enforce their own authorization rules based on token claims such as scopes or roles.

## Example Request Flow
Browser → Login redirect → OpenID Provider
OpenID Provider → Authorization Code → 
BFF → Token endpoint → Tokens received
BFF → Session cookie → Browser

Browser → Request with cookie →
BFF → Access token → Microservice
Microservice → Verify token → Return response


## Alternative: SPA Token-Based Approach

An alternative architecture is a **Single Page Application (SPA)** where the browser handles the OIDC flow directly.

In this approach, the frontend redirects the user to the OpenID Provider and receives an **authorization code** after login. The SPA exchanges this code for tokens using the **Authorization Code Flow with PKCE**. The resulting **access token** is then stored in the browser (often in memory or local storage) and sent directly to backend APIs in the `Authorization` header.


### Advantages

- Simpler architecture with fewer backend components  
- Highly scalable since APIs are called directly from the browser    

### Disadvantages

- Tokens are stored in the browser and may be exposed through **XSS attacks**
- Token refresh logic must be handled in the frontend
- APIs must trust tokens coming directly from the browser

For applications requiring stronger security controls, a **Backend-for-Frontend (BFF)** architecture is often preferred because tokens remain on the server and are never exposed to the browser.
